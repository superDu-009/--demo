## 1. 接口设计合理性

### 1.1 [P0] 分镜列表缺少分页参数

**问题描述**: `GET /api/scene/{sceneId}/shots` 响应返回全量数组，无分页支持。一个分场可能有几十甚至上百个分镜，全量返回会导致首屏加载慢、内存占用高。

**前端影响**: 分镜工作台需要虚拟滚动或分页渲染，无分页参数意味着无法做懒加载。

**修改建议**:
```
GET /api/scene/{sceneId}/shots?page=1&size=20&status=2(可选)
```
响应结构增加 `total`、`hasNext` 字段，或统一使用 `PageResult<ShotVO>` 包装。

### 1.2 [P0] 项目列表分页参数响应缺少标准字段

**问题描述**: `GET /api/project/list` 响应返回 `{total, list}`，但前端标准分页组件通常还需要 `page`、`size`、`hasNext` 等字段来驱动翻页器。

**修改建议**: 统一使用 `PageResult` 包装，结构如下：
```json
{
  "code": 0,
  "data": {
    "total": 100,
    "page": 1,
    "size": 10,
    "hasNext": true,
    "list": [...]
  }
}
```

### 1.3 [P1] 分镜列表响应已关联资产信息，但缺少项目级聚合查询

**问题描述**: `GET /api/scene/{sceneId}/shots` 的响应中包含了 `assetRefs`（内含 `assetName`、`primaryImage`），说明后端做了 JOIN 查询。但前端分镜工作台经常需要「按项目维度」查看所有分镜状态（不分场），当前无此接口。

**修改建议**: 增加 `GET /api/project/{projectId}/shots?sceneId=xxx(可选)&status=xxx(可选)` 接口，支持跨分场的分镜聚合查询。

### 1.4 [P1] 批量审核接口缺少前置校验信息

**问题描述**: `POST /api/shot/batch-review` 请求体中 `shotIds` 直接传 ID 数组，但没有返回「哪些 ID 审核成功/失败」的明细。前端需要知道批量操作中哪些成功、哪些因状态不对被跳过。

**修改建议**: 响应体增加操作结果明细：
```json
{
  "code": 0,
  "data": {
    "totalCount": 10,
    "successCount": 8,
    "failedCount": 2,
    "failedDetails": [
      { "shotId": 123, "reason": "分镜状态为已打回，无法再次打回" }
    ]
  }
}
```

### 1.5 [P2] 资产列表缺少分页

**问题描述**: `GET /api/project/{projectId}/assets` 返回全量数组。当项目资产较多时（50+ 角色/场景），全量加载会影响性能。

**修改建议**: 增加分页参数 `page`、`size`，以及按 `status` 过滤的参数。

### 1.6 [P2] 缺少分镜「重新生成」接口

**问题描述**: 分镜打回后需要重新生成图片/视频，但文档中只有 `PUT /api/shot/{id}` 更新分镜字段。前端需要一个明确的「重新生成」语义接口来触发 AI 重新出图，同时增加 `generation_attempts` 计数。

**修改建议**: 增加 `POST /api/shot/{shotId}/regenerate` 接口，后端内部将 `generation_attempts + 1`，`status` 重置为 `1-生成中`，并触发 AI 任务。

### 1.7 [P2] 缺少「撤销确认」资产的能力

**问题描述**: 资产有 `confirm` 接口（status → 1），但没有反操作。前端资产编辑页误点确认后无法回退。

**修改建议**: 增加 `PUT /api/asset/{id}/unconfirm` 或将确认逻辑合并到 `PUT /api/asset/{id}` 中，允许修改 `status` 字段。

### 1.8 [P2] 接口路径风格不一致

**问题描述**: 资产列表使用 `/api/project/{projectId}/assets`（嵌套路径），但资产更新使用 `/api/asset/{id}`（扁平路径）。前端路由映射和权限校验容易混淆。

**修改建议**: 统一风格。建议资产相关全部使用嵌套路径或全部使用扁平路径，保持一致性。例如：
- `GET /api/project/{projectId}/assets` → 列表
- `PUT /api/project/{projectId}/assets/{id}` → 更新
- `DELETE /api/project/{projectId}/assets/{id}` → 删除

### 1.9 [P2] 缺少批量排序接口

**问题描述**: 分镜工作台支持拖拽排序（调整 `sort_order`），但当前只能通过 `PUT /api/shot/{id}` 逐个更新。拖拽排序后批量保存是刚需。

**修改建议**: 增加 `PUT /api/scene/{sceneId}/shots/sort` 接口，请求体：
```json
{ "shots": [{ "id": 1, "sortOrder": 0 }, { "id": 2, "sortOrder": 1 }] }
```

## 2. 数据结构适配

### 2.1 [P1] `shot_asset_ref` 拆分合理，但需要批量查询优化

**问题描述**: `shot_asset_ref` 作为独立关联表在数据库范式上是正确的，但从前端角度看，查询单个分镜详情时需要 JOIN `asset` 表获取 `name` 和 `reference_images[0]`。当前设计在 `listShots` 中已包含 `assetRefs`（含 `assetName`、`primaryImage`），说明后端理解这个 JOIN 需求。

**潜在问题**: 如果前端需要「查询某个资产被哪些分镜引用」来展示资产依赖关系，当前 `idx_asset_id` 索引可以支撑反向查询，但缺少对应接口。

**修改建议**: 增加 `GET /api/asset/{assetId}/references` 接口，返回引用该资产的分镜列表，用于资产详情页展示「引用关系」。

### 2.2 [P1] `shot` 表缺少 `ai_task_id` 关联字段

**问题描述**: `ai_task` 表有 `shot_id` 反向关联，但 `shot` 表没有记录当前正在执行的 AI 任务 ID。前端分镜卡片需要显示「生成中」状态并关联具体的 AI 任务进度，当前只能通过 `GET /api/ai/task/{taskId}` 单独查询，但前端不知道该查哪个 `taskId`。

**修改建议**: 在 `shot` 表中增加 `image_task_id` 和 `video_task_id` 字段，或者在 `GET /api/scene/{sceneId}/shots` 响应中直接内嵌当前 AI 任务状态：
```json
{
  "id": 1,
  "status": 1,
  "currentAiTask": {
    "taskId": 456,
    "taskType": "image_gen",
    "status": 1,
    "estimatedSeconds": 30
  }
}
```

### 2.3 [P2] `episode` / `scene` 缺少 `episode_id` 到 `project_id` 的快速查询

**问题描述**: `scene` 表只有 `episode_id` 索引，没有 `project_id` 冗余字段。前端经常需要「查询项目下所有分场」或「查询项目下所有分集的分场」，需要多层 JOIN。

**修改建议**: 在 `scene` 表增加 `project_id` 冗余字段（或至少在响应 VO 中透出），减少前端跨层查询的复杂度。

### 2.4 [P2] 时间字段格式不统一

**问题描述**: 项目列表响应中 `createTime` 标注为 `ISO-8601`，但其他接口的响应示例中没有统一标注。部分字段如 `workflow/status` 中完全没有时间戳。

**修改建议**: 全局约定所有时间字段使用 ISO-8601 格式（如 `2026-04-19T16:30:00+08:00`），并在文档中明确。前端统一使用 `dayjs` 或 `date-fns` 处理。

### 2.5 [P2] `reference_images` 字段类型前后端不一致风险

**问题描述**: `asset` 表中 `reference_images` 是 JSON 类型存储 URL 数组，但 `shot` 表中直接用 `generated_image_url`（VARCHAR）存储单张图片。这意味着前端在分镜列表中无法看到历史生成记录（每次生成覆盖前一张图）。

**修改建议**: 如果产品需要保留分镜历史版本图片，应将 `generated_image_url` 改为 JSON 数组或单独建 `shot_version` 表。当前 `version` 字段仅记录版本号但没有版本历史表支撑，前端无法做「查看历史版本」功能。

## 3. 流程引擎 — 轮询接口是否足够支撑进度条

### 3.1 [P0] `progress` 字段精度不足

**问题描述**: 当前 `GET /api/project/{id}/workflow/status` 响应：
```json
{
  "progress": "已完成步骤数 / 总步骤数",
  "steps": [...]
}
```
`progress` 是「已完成步骤数 / 总步骤数」，粒度是「步骤级」。但前端进度条需要展示：
- 每个步骤内部的子任务进度（如：生图进度 30%、生视频进度 60%）
- AI 任务级别的实时进度（Seedance 任务通常有百分比进度）
- 分镜级别的统计进度（如：100 个分镜，已完成 30 个）

**当前结构无法区分**：
- 步骤正在执行中 vs 步骤已完成（只能看 `status`，但没有百分比）
- 多分镜并行处理时的整体完成比例

**修改建议**:
```json
{
  "executionLock": 1,
  "currentStep": "image_gen",
  "overallProgress": 45,       // 整体百分比 0-100
  "totalShots": 100,           // 总待处理分镜数
  "processedShots": 45,        // 已处理分镜数
  "estimatedRemainingSeconds": 180,  // 预估剩余时间
  "steps": [
    {
      "stepType": "image_gen",
      "status": 1,
      "progress": 60,          // 步骤内部进度百分比
      "currentDetail": "正在生成第 45/100 个分镜",  // 用户可读描述
      "errorMsg": null
    }
  ]
}
```

### 3.2 [P1] 缺少 SSE/WebSocket 替代方案

**问题描述**: 前端 3s 轮询在流程执行期间（可能持续数分钟到数小时）会产生大量无效请求。如果后端支持 SSE（Server-Sent Events）或 WebSocket，前端可以订阅实时进度推送，体验远优于轮询。

**修改建议**: MVP 阶段保留轮询，但预留 SSE 接口 `GET /api/project/{id}/workflow/stream`，EventStream 格式推送进度更新。

### 3.3 [P1] `workflow/status` 缺少当前执行的分集信息

**问题描述**: 流程引擎按分集顺序执行，但 `workflow/status` 响应中没有当前正在处理哪个分集（`episode_id`）的信息。前端无法在进度条旁展示「正在处理第 2 集 / 共 5 集」。

**修改建议**: 在响应中增加 `currentEpisodeId` 或 `currentEpisodeTitle` 字段。

## 4. 上传交互 — TOS 预签名直传流程

### 4.1 [P0] 缺少上传完成回调/通知机制

**问题描述**: 当前流程：
1. 前端 `POST /api/tos/presign` → 获取 `uploadUrl`
2. 前端 `PUT uploadUrl` 直传 TOS
3. **❌ 后端不知道上传是否完成**

前端直传 TOS 后，后端数据库中没有对应记录。以小说上传为例：前端拿到 `accessUrl` 后，需要再调一次 `PUT /api/project/{id}` 把 `novelTosPath` 设置进去。这个两步操作不是原子性的——如果前端第二步失败，文件已上传但项目没有关联路径。

**修改建议**: 增加上传完成通知接口：
```
POST /api/tos/complete
{
  "key": "projects/123/novel_xxx.txt",  // TOS 中的 object key
  "projectId": 123,                      // 关联的项目ID
  "fileType": "novel",                   // 文件类型标识
  "metadata": { "originalName": "xxx.txt", "size": 12345 }
}
```
后端收到后校验文件确实存在于 TOS（HEAD 请求），然后更新数据库记录。

或者采用更优雅的方案：前端在 presign 时传入 `callbackUrl`，TOS 上传完成后回调后端（火山 TOS 支持上传回调功能）。

### 4.2 [P1] 缺少上传进度支持

**问题描述**: `PresignResult` 中没有 `maxFileSize` 限制信息。前端需要在上传前告知用户文件大小限制，并在上传过程中展示进度条。

**修改建议**: 在 `PresignResult` 中增加 `maxFileSize` 字段：
```json
{
  "uploadUrl": "...",
  "accessUrl": "...",
  "expiresIn": 3600,
  "maxFileSize": 52428800  // 50MB
}
```

### 4.3 [P1] 缺少分片上传支持

**问题描述**: 如果用户上传的小说文件 > 50MB（配置中 `max-file-size: 50MB`），当前方案无法处理。视频文件尤其容易超限。

**修改建议**: 对于大文件（>10MB），提供分片上传方案：
- `POST /api/tos/presign-multipart` 获取分片列表
- 前端逐片上传
- `POST /api/tos/complete-multipart` 合并

### 4.4 [P2] 预签名接口鉴权标注矛盾

**问题描述**: Controller 表格中标注鉴权为 `否*`，注释又写「预签名接口需登录态」。表格和注释矛盾，前端容易困惑。

**修改建议**: 修正表格为「是」，并删除星号注释。

## 5. 错误码覆盖度

### 5.1 [P1] 缺少前端高频交互场景的细分错误码

**当前错误码**:

| 范围 | 模块 | 示例 |
|------|------|------|
| 40000-40099 | 参数校验 | 40001 用户名或密码错误 |
| 40100-40199 | 认证鉴权 | 40100 未登录, 40101 Token过期 |
| 40300-40399 | 权限不足 | 40300 无操作权限 |
| 40400-40499 | 资源不存在 | 40400 项目不存在 |
| 40900-40999 | 业务冲突 | 40900 项目正在执行中 |
| 50000-50099 | 系统异常 | 50000 服务器内部错误 |
| 51000-51099 | AI服务异常 | 51000 AI调用失败 |
| 51100-51199 | 存储异常 | 51100 TOS上传失败 |

**前端需要但未覆盖的场景**:

| 错误码 | 场景 | 前端交互 |
|--------|------|----------|
| `40002` | 字段校验失败（具体哪个字段） | 表单高亮对应字段 |
| `40003` | 文件类型不支持 | Toast 提示允许的类型 |
| `40004` | 文件大小超限 | Toast 提示最大限制 |
| `40005` | 预签名 URL 已过期 | 自动重新请求 presign |
| `40301` | 非项目创建者无权操作 | 跳转无权限页 |
| `40901` | 资产已被分镜引用，不可删除 | Toast 提示引用关系 |
| `40902` | 分镜状态不支持当前操作 | 禁用对应按钮 |
| `42900` | 请求频率过高（AI 调用限流） | 倒计时后重试 |
| `51001` | AI 模型超时/排队中 | 展示排队状态 |
| `51002` | AI 生成内容为空/不合格 | 提示重新生成 |
| `51101` | TOS 存储空间不足 | 联系管理员 |

**修改建议**: 补充上述错误码到 2.2 节的错误码规范表中，特别是 `40002-40005`（前端表单校验和上传场景）和 `40901-40902`（业务状态冲突）是前端高频处理场景。

### 5.2 [P1] 缺少全局错误处理约定

**问题描述**: 文档定义了 `Result<T>` 和错误码，但没有说明前端如何统一处理：
- `40100/40101` → 自动跳转登录页？
- `40300` → Toast 提示还是跳转无权限页？
- `50000` → 展示全局错误页？

**修改建议**: 在文档中增加「前端错误处理约定」小节，明确各错误码范围对应的前端行为。

### 5.3 [P2] HTTP 状态码与业务码的映射关系不明确

**问题描述**: 所有接口都返回 HTTP 200 + `Result<T>` 包装的业务码。但前端 `axios` 拦截器通常基于 HTTP 状态码做统一处理。需要明确：
- 401/403 场景是否仍然返回 HTTP 200 + 业务码 40100/40300？
- 还是返回对应 HTTP 状态码？

**修改建议**: 统一约定：鉴权相关（401xx、403xx）返回对应 HTTP 状态码（401/403），其他业务错误统一返回 HTTP 200 + 业务码。

## 6. 其他前端相关建议

### 6.1 [P2] 缺少大列表的「搜索/排序」接口支持

分镜工作台、资产列表等页面需要按名称搜索、按创建时间排序。当前所有列表接口只支持 `assetType` 过滤，缺少 `keyword` 搜索和 `sortBy` 排序参数。

### 6.2 [P2] `stylePreset` 字段结构需明确

`stylePreset` 在 `project` 和 `asset` 表中都作为 JSON 存储，但结构仅在 workflow 配置中给出示例 `{ artStyle, colorTone }`。前端风格预设组件需要明确的字段定义，建议增加枚举或约束说明。

### 6.3 [P2] 建议增加 ETag/Last-Modified 支持

分镜图片和视频 URL 可能因重新生成而变化，建议后端在资源 URL 变化时更新，前端通过 ETag 做缓存失效控制。

## 总结

老克的系分文档整体结构清晰，技术选型合理，模块划分明确。但从前端开发视角来看，存在以下核心改进方向：

1. **分页缺失**: 分镜列表、资产列表等核心列表接口均无分页，直接影响工作台性能。
2. **流程进度太粗**: 步骤级进度无法支撑 AI 任务级别的实时进度条，需要增加百分比和预估时间。
3. **上传闭环不完整**: 预签名直传后缺少后端感知机制，存在数据不一致风险。
4. **错误码需细化**: 前端表单校验、上传校验、业务冲突等高频场景缺少细分错误码。

建议在进入开发前，先就上述 P0 问题与后端对齐方案，P1 问题在 Sprint 1-2 中逐步完善。
