## 一、评审结论说明

当前测试用例文档中「第 13 节 前端集成测试」仅包含 **8 条用例**（6 P0 + 2 P1），与前端系分中描述的 **7 个核心页面 + 6 个 Store + 3 个 Composable + 7 个 API 模块 + 路由守卫 + 拦截器 + 安全策略** 的复杂度严重不匹配。

存在以下核心问题：
1. 多处用例描述与系分设计**不一致**（如虚拟滚动 vs 分页、固定 3s 轮询 vs 三档退避）
2. **大量关键前端场景完全缺失**（路由守卫、Pinia 状态同步、TOS 直传完整流程、拖拽边界、轮询超时保护等）
3. 用例粒度过粗，多数用例仅描述了「正常路径」，缺乏**边界条件、异常场景、交互细节**

**结论：不通过。建议大幅扩充前端测试用例后重新提交评审。**

## 三、缺失的前端关键场景

### P0 级缺失（阻塞性）

#### P0-1: 路由守卫测试 — 完全缺失

系分 `router/guards.ts` 明确定义了以下守卫逻辑，但测试用例中**一条都没有**：

| 场景 | 预期行为 |
|------|----------|
| 未登录访问 `/projects` | 跳转 `/login?redirect=/projects` |
| 已登录访问 `/login` | 跳转 `/projects` |
| 登录后带 redirect 参数 | 跳回 redirect 指定路径 |
| 访问不存在的 `/unknown-path` | 渲染 NotFound 组件 |
| Token 从 localStorage 恢复 | 页面刷新后 authStore 自动恢复登录态 |

#### P0-2: Pinia 状态同步与持久化 — 完全缺失

| 场景 | 预期行为 |
|------|----------|
| authStore token 持久化 | setToken 同时写入 localStorage（24h TTL） |
| clearAuth 清理 | token + userInfo 清空 + localStorage 移除 |
| 页面刷新后 store 恢复 | 从 localStorage 读取 token 恢复登录态 |
| 跨组件状态同步 | 一个组件修改 store，其他组件响应式更新 |
| assetStore 按类型分组 | fetchAssets 按 asset_type 正确分组到 assetsByType |
| workflowStore 轮询状态 | startExecution → isRunning 变为 true |

#### P0-3: Token 过期与 401 处理 — 用例描述错误

- TC-SEC-002 仅说"返回 401，要求重新登录"，**未描述前端行为**
- 系分明确：401 → `authStore.clearAuth()` → `ElMessage.error('登录已过期')` → `router.push({ name: 'Login', query: { redirect } })`
- **缺少测试：前端收到 401 后是否正确清除状态、展示提示、跳转登录页**

#### P0-4: TOS 直传完整流程 — 测试覆盖不足

TC-TOS-002 仅描述"前端使用预签名 URL PUT 上传文件 → 文件上传成功"，但系分 `useTosUpload` composable 的完整流程是：

1. 文件校验（大小 + 类型）
2. 获取预签名 URL（POST /tos/presign）
3. XHR PUT 直传 TOS（带上传进度）
4. 通知后端（POST /tos/complete）
5. complete 失败重试 1 次
6. 预签名过期（40005）自动重试

**以下子场景完全缺失：**
- 文件大小超限校验（> 50MB）
- 文件类型不支持校验
- 上传进度展示（progress ref）
- complete 失败重试逻辑
- 预签名 URL 过期自动重试
- 上传中 loading 状态阻断重复提交

#### P0-5: 任务轮询三档退避机制 — 用例描述错误

TC-FE-006 写"3s 轮询更新状态"，但系分明确使用**三档退避**：

| 轮询次数 | 间隔 | 说明 |
|----------|------|------|
| 1-3 次 | 3s | 即时反馈 |
| 4-10 次 | 10s | 降频 |
| 11+ 次 | 30s | 封顶 |

同时缺失：
- 超时保护（maxPolls=2400，约 2h）
- onDone 回调触发
- 组件卸载自动停止轮询（onUnmounted）
- 工作流轮询（useWorkflowPolling）完全未覆盖

## 六、总体评价

| 维度 | 评分 | 说明 |
|------|------|------|
| 用例完整性 | 2/10 | 8 条用例覆盖不到前端功能的 30%，大量关键场景缺失 |

| 优先级分配 | 5/10 | P0/P1 基本合理，但缺失的用例中很多应该是 P0 |
| 测试分层 | 5/10 | 第 13 节定位为"前端集成测试"，但多数用例更像 E2E 验收测试，缺少组件级集成测试 |

### 核心问题总结

1. **虚拟滚动 vs 分页**：TC-FE-005 描述与系分 v1.1 决策完全矛盾，系分已明确移除虚拟滚动改为分页模式
2. **轮询机制错误**：TC-FE-006 写"3s 轮询"，实际是三档退避 3s→10s→30s
3. **路由守卫零覆盖**：作为前端核心安全机制，一条测试都没有
4. **Pinia 状态管理零覆盖**：状态同步、持久化、恢复均未测试
5. **TOS 直传不完整**：仅覆盖成功路径，缺少文件校验、重试、进度等
6. **工作流轮询零覆盖**：useWorkflowPolling composable 完全没被测试
7. **拖拽边界零覆盖**：SortableJS 仅一句话描述
8. **错误处理太笼统**：7 种 ERROR_ACTION 类型没有拆分测试

## 七、改进建议

1. **修正与系分不一致的描述**：立即修改 TC-FE-005（虚拟滚动→分页）、TC-FE-006（3s轮询→三档退避）
2. **按前端系分章节逐条补充用例**：建议按 4.1~4.7 页面 + 5.1 拦截器 + 7 状态管理 + 8 Composables 分别编写用例
3. **区分集成测试和 E2E 测试**：第 13 节建议定位为组件/API 集成测试，端到端流程放在第 14 节
4. **补充前置条件和验证步骤**：当前用例缺少数据准备、环境配置、具体验证方式（如"检查 localStorage 中 token 字段"）
5. **补充异常/边界场景**：每个正常用例至少补充 1-2 个异常场景
